# where;_____ 분할 렌더 누락 버그 — 인수인계서

> 작성일: 2026-06-28
> 대상: 다음 claude.ai 세션 / Claude Code 협업자
> 한 줄 요약: **긴 음악영상(분할 렌더)에서 where;_____ 로고·이퀄·텍스트가 통째로 사라지던 버그. 원인이 4겹이었고 전부 해결됨.**

---

## 0. 증상 (당시 상황)

- **짧은 영상(23분 이하, 1~3곡, 단일샷)** → where;_____ 정상
- **긴 영상(39분+, 10곡+, 분할 렌더)** → where;_____ 로고·이퀄라이저·텍스트가 **통째로 없음**. 배경 이미지 + 음악만 나옴
- 음원은 정상 재생됨
- 곡 수가 많을수록(청크 분할 경로) 누락

핵심 혼란 포인트: "로고만 안 나온다"가 아니라 **Remotion이 그리는 모든 것이 사라짐**. 이유는 Remotion 렌더가 실패하면 ffmpeg 폴백(배경+오디오만 합성)으로 떨어지기 때문. 폴백 영상엔 Remotion 오버레이가 0.

---

## 1. 아키텍처 맥락 (알아야 할 것)

- 음악영상은 **Remotion**으로 렌더. 컴포넌트 = `travel-pipeline/remotion/src/MusicViz.tsx`, 진입 = `render.mjs`
- 긴 영상은 **청크 분할 렌더**: `_CHUNK_MAX_SEC`(현재 env `MUSIC_CHUNK_MAX_SEC=420`=7분) 초과 시 여러 청크로 쪼개 **AWS Lambda 병렬 렌더** → ffmpeg로 concat → 풀 오디오 mux
- 관련 파일: `travel-pipeline/services/music_video.py` (`_render_chunks`, `_render_remotion`, `make_video`)
- 청크는 `muted=True`로 **비디오만** 렌더(오디오는 나중에 풀 mix를 mux로 입힘)
- Remotion 렌더 실패 시 → **ffmpeg 폴백**(where 없는 영상)으로 떨어지는 구조

## 2. 4겹의 원인 (순서대로 한 겹씩 드러났음)

### 원인 ① where_text 빈 값 (초기 오진 포함)
- `MusicViz.tsx` L241: `labelText = (designConfig?.where_text || "").trim() || locationEn`
- **상단 라벨**은 `where_text`를 쓰고, **중앙 메인 로고**는 `playlist_text`를 씀 (서로 다른 필드!)
- 추적 초기에 중앙 로고가 안 보이길래 `where_text`를 원인으로 **오진**, Supabase에서 `where_text`에 `where;_____`를 넣었음 → 이게 **상단 라벨에 where;_____를 박는 부작용**을 냈음 (나중에 비워서 원복)
- 실제 중앙 로고(`playlist_text`)는 처음부터 정상. 중앙 로고가 안 보인 건 ②③④로 인한 **렌더 실패(폴백)** 때문이었음

### 원인 ② 폰트 과다 로딩 → 브라우저 crash
- `MusicViz.tsx`의 `loadFont()` 들이 **weight·subset 무제한**으로 호출 → CJK 폰트(Noto Serif KR)가 **992개 네트워크 요청** 생성 → 헤드리스 브라우저 메모리 폭발 → `Page crashed!`
- 해결: 모든 폰트에 `loadFont("normal", { weights:[...], subsets:[...] })` 적용. CJK는 weight 1개(`["400"]`) + `subsets:["korean"]` + `ignoreTooManyRequestsWarning:true`. 영어는 `subsets:["latin"]` + 필요한 weight만
- 관련 PR: #276, #277

### 원인 ③ Lambda 메모리·디스크 부족
- 폰트 줄여도 `Page crashed!` 지속 → 로그에서 `Lambda 실패 ... Sandbox.Failure` 발견
- Lambda 함수가 **mem2048mb / disk2048mb** = 2GB로는 긴 청크 렌더에 부족 → `Sandbox.Failure`(메모리) / `Runtime.TruncatedResponse`(디스크)
- 해결: **Remotion Lambda 함수를 4GB로 재배포**
  - 명령(remotion 폴더에서): `npx remotion lambda functions deploy --memory=4096 --disk=4096 --timeout=900`
  - 새 함수명: `remotion-render-4-0-481-mem4096mb-disk4096mb-900sec`
  - Railway env `REMOTION_LAMBDA_FUNCTION_NAME`을 새 함수명으로 변경
  - AWS 자격증명은 Railway Variables의 AWS_ACCESS_KEY_ID/SECRET/REGION(us-west-2). Claude Code 환경엔 없어서 대표가 로컬 PowerShell에서 직접 배포함
  - 4GB는 유지(2GB로 되돌리면 Sandbox.Failure 재발). 메모리·디스크가 실제 원인이었음

### 원인 ④ 청크 오디오 메타데이터 타임아웃
- 4GB 후 `Sandbox.Failure`는 사라졌으나 새 폴백: `delayRender() "Waiting for audio metadata ... not cleared after 28000ms"`
- 원인: 청크는 muted인데도 `MusicViz.tsx`의 `useAudioData(audioSrc)` + `<Audio>`가 음원(mp3) 메타데이터를 R2에서 로딩 → 여러 청크가 동시에 같은 mp3 호출 → 28초 타임아웃
- 해결 (PR #278):
  - `MusicViz.tsx`: `muted` prop 추가. `audioData = muted ? null : rawAudioData` (결과 무시 → 이퀄은 사인파 폴백). `{!muted && <Audio src={audioSrc} />}` (Audio 미렌더)
  - `render.mjs`: `muted`를 inputProps로 전달. 로컬 `delayRenderTimeoutInMilliseconds:120000`, Lambda `timeoutInMilliseconds:120000`

## 4. 남은 리스크 — 2시간 영상(30곡)

- 10곡(6~7청크)은 검증됐으나 **2시간 영상 = 약 30곡 = 약 17청크**는 아직 미검증
- 청크가 2.5배 많아져 **R2 동시 호출 증가** → 120초 타임아웃이 또 터질 수 있음
- 그 경우 해결책(옵션2): **청크에서 `useAudioData`를 완전히 제거**. muted일 때 무음 더미 오디오(번들 포함)로 대체해 R2 호출 자체를 없앰 → 청크 수 무관하게 안전
- 2시간 영상 처음 뽑을 때 폴백/`audio metadata` 로그 나오면 이 방향으로

## 5. 디버깅 방법론 (다음 사람에게 유용)

- **추측 작업지시서 반복 금지.** 로그 찍어 직접 비교가 빠름. 이번에 결정적이었던 건:
  1. props-check 로그로 "props는 정상 전달됨" 확인 (where_text, show_playlist 등)
  2. debug-chunk로 **청크 mp4를 R2에 올려 육안 확인** → 성공 청크엔 where 있고, 실패 청크는 404(애초에 안 만들어짐)임을 확인 → "청크 일부가 렌더 실패 후 폴백" 확정
  3. Railway 로그에서 `폴백`/`Sandbox.Failure`/`Runtime.TruncatedResponse`/`audio metadata` 검색으로 **실패 원인이 한 겹씩 바뀌는 것** 추적
- **로그 검색 키워드**: `폴백`, `REMOTION_OK`, `청크`, `Sandbox.Failure`, `TruncatedResponse`, `audio metadata`, `network requests`
- **CloudWatch**: us-west-2(오레곤), 로그그룹 `/aws/lambda/remotion-render-4-0-481-mem*`. Lambda 내부 실패는 여기서 봄
- **Railway 무료 로그 보존 짧음** → 시간범위 좁혀 검색. 한국시간 UTC+9

## 6. 핵심 교훈

- where;_____는 **3개 레이어**가 얽혀 있음: ① 상단 라벨(`where_text` → 폴백 locationEn) ② 중앙 로고(`playlist_text`) ③ Remotion 렌더 성공 여부. 안 보일 때 어느 레이어 문제인지부터 구분할 것
- **Remotion 렌더 실패 = ffmpeg 폴백 = 모든 오버레이 사라짐.** "특정 요소만 안 나온다"가 아니라 "전부 사라졌다"면 렌더 자체 실패를 의심
- 짧은 영상 정상 + 긴 영상만 실패 = **분할 렌더(청크) 경로의 리소스 문제** 우선 의심
- Claude Code "이미 반영됨" 보고는 신뢰 말고 **curl raw.githubusercontent.com으로 직접 검수**
